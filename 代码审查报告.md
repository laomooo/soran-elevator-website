## 1. 项目结构与架构

### 1.1 目录组织

```
soran-website/
├── .github/workflows/deploy.yml   # GitHub Actions 部署流水线
├── supabase/
│   ├── schema.sql                 # 完整数据库初始化
│   ├── migration-add-category-image.sql
│   └── migration-add-business-services.sql
├── public/                        # logo、.nojekyll 等静态资源
├── scripts/                       # 测试脚本
├── src/
│   ├── app/
│   │   ├── layout.jsx             # 根布局
│   │   ├── globals.css            # 全局样式
│   │   ├── (site)/                # 前台路由组（共享 Navbar/Footer）
│   │   └── admin/                 # 后台管理路由组（独立鉴权布局）
│   ├── components/                # 可复用组件
│   └── lib/
│       ├── supabase.js            # Supabase 客户端封装
│       ├── data.js                # 服务端数据获取（含兜底数据）
│       └── client-data.js         # 客户端数据获取（解决 SSG 不同步）
```

**评价**：目录结构清晰，前后台通过路由组 `(site)` / `admin` 隔离，职责分明。数据层拆分为 `data.js`（服务端/构建时）和 `client-data.js`（浏览器端），有效解决了 Next.js 静态导出与 Supabase 动态数据之间的矛盾。

### 1.2 技术栈选型

| 技术 | 评价 |
|------|------|
| Next.js 14 静态导出 | 适合 GitHub Pages 部署，构建产物可直接托管；SSG 与 Supabase 实时刷新通过客户端组件补充 |
| Tailwind CSS | 原子化样式，移动优先响应式实现良好 |
| Supabase | 作为 BaaS 提供数据库、Auth、Storage，降低后端维护成本 |
| GitHub Actions | 自动化部署，配置简洁 |

## 2. 安全性审查

### 2.1 认证与授权

- ✅ 后台使用 Supabase Auth `signInWithPassword` 进行邮箱密码登录。
- ✅ 所有内容表均启用 RLS，且策略为「anon 只读、authenticated 可写」。
- ✅ `inquiries` 表限制 anon 仅可插入，认证用户可读/改/删。
- ✅ 源码中未发现 `service_role` key 的硬编码或误用。
- ⚠️ 建议：后台管理页面可进一步增加权限角色校验（如仅允许特定邮箱），当前所有 authenticated 用户均可管理。

### 2.2 环境变量与密钥

- ✅ `.env.local` 已正确加入 `.gitignore`。
- ✅ `.env.example` 仅包含占位符，无真实密钥。
- ✅ GitHub Actions 通过 `secrets` / `vars` 注入环境变量。

### 2.3 输入验证

- ✅ `ContactForm` 对姓名、电话、邮箱、留言内容做了前端校验。
- ⚠️ 后台表单主要依赖数据库约束和 RLS，建议增加 slug 唯一性、必填字段等更明确的前端校验与错误提示。

### 2.4 图片上传

- ✅ 图片上传限制为图片类型和 5MB 大小。
- ⚠️ 上传文件命名使用随机字符串，可避免覆盖，但无文件名后缀白名单限制（已通过 `accept` 限制前端选择）。
- ⚠️ 删除记录时未同步删除 Storage 中的图片，存在垃圾文件风险。

## 3. 性能审查

### 3.1 静态导出

- ✅ 所有前台页面均为静态生成，首屏加载快。
- ✅ `next.config.js` 已设置 `images.unoptimized: true`，适配静态导出和外部 Storage URL。

### 3.2 客户端数据获取

- ✅ 首页核心数据、产品列表、新闻列表、产品/新闻详情、主营业务等均使用客户端组件在挂载后拉取最新 Supabase 数据。
- ⚠️ 客户端请求较多，可考虑在 `client-data.js` 中增加 SWR/React Query 缓存，减少重复请求。

### 3.3 图片优化

- ⚠️ 当前使用 `next/image` 但关闭了优化（`unoptimized: true`），图片未经过自动压缩、WebP 转换、响应式尺寸。
- 建议：上线前对上传图片进行压缩（如限制最大宽度 1200px），或考虑接入 CDN 图片处理服务。

### 3.4 包体积

- 构建产物首屏 JS 约 87.2 kB 共享 chunk + 页面 chunk，体积可控。
- 建议：定期检查 `next bundle-analyzer` 输出，避免不必要的依赖。

## 5. 数据库设计审查

### 5.1 表结构

当前 9 张表：
- `product_categories`：产品分类，含 `image_url`
- `products`：产品，含 `images` 数组、`specs` JSONB
- `solutions`：解决方案
- `cases`：客户案例
- `news`：新闻动态，含 `cover_image`
- `inquiries`：在线留言
- `company_info`：公司信息键值对
- `site_settings`：站点设置键值对
- `business_services`：主营业务，含 `image_url`

**评价**：表结构覆盖官网全部模块，字段命名统一为 snake_case，主键使用 UUID，含 `created_at` / `updated_at`。

### 5.2 索引与约束

- ✅ 所有业务表对 `slug` 设置唯一约束。
- ✅ 外键 `products.category_id` 使用 `on delete set null`。
- ⚠️ 建议：对高频查询字段增加索引，如 `products.is_featured`、`news.is_published`、`news.published_at`、`business_services.sort_order`。

### 5.3 RLS 策略

- ✅ 所有表均启用 RLS。
- ✅ Storage bucket 已配置公开读、认证写策略。
- ⚠️ 建议：对 `company_info` / `site_settings` 等只允许读取， authenticated 写策略当前为 `for all`，权限较宽，可接受但需注意管理员账号安全。
