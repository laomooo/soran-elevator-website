# 广州索然电梯配件官网 · AI 上下文管理方案

**制定日期**：2026-07-10  
**适用对象**：本项目所有 AI 协作会话  
**目标**：确保项目知识在会话之间不丢失，AI 能够快速理解项目背景、技术约束、品牌规范，避免重复沟通成本。

---

## 1. 为什么需要上下文管理

本项目是一个持续迭代的企业官网工程，已历经多轮需求变更：
- 从 AI 提示词文档到完整代码实现
- 从静态页面到 Supabase 动态数据
- 从基础展示到 Apple 风格动效
- 从产品/新闻图片到分类/主营业务图片

每一轮变更都积累了关键决策（技术栈、数据模型、视觉规范、已知问题）。如果这些信息散落在聊天历史中，新会话需要重新询问和确认，效率低下且容易出错。因此需要一套结构化的上下文管理机制。

## 3. 记忆文件维护规则

### 3.1 每日日志（.workbuddy/memory/YYYY-MM-DD.md）

**何时更新**：
- 每次完成实质性工作后立即追加
- 修复 bug、新增功能、修改配置、部署验证

**记录内容**：
- 完成了什么（功能/修复/优化）
- 修改了哪些关键文件
- 构建验证结果
- dev server 运行端口
- 当前阻塞或待确认事项

**不记录**：
- 搜索过程、临时报错、中间文件路径
- 已通过修复解决的问题（只记录结果）

### 3.2 项目长期记忆（.workbuddy/memory/MEMORY.md）

**何时更新**：
- 数据模型变更（新增表、字段）
- 技术栈或部署方式变更
- 品牌规范调整
- 重大功能交付
- 用户偏好或项目约定固化

**维护方式**：
- 按主题分块（项目概述、技术栈、数据模型、品牌规范、已交付、待办）
- 每次新增内容合并到已有块，不重复堆砌
- 定期（如每月）整理旧日志，提炼到 MEMORY.md

### 3.3 跨项目用户偏好（~/.workbuddy/MEMORY.md）

**何时更新**：
- 用户明确表达的个人习惯（如「我喜欢表格化」、「先确认再执行」）
- 跨项目通用的技术偏好（如「偏好静态托管+云数据库」）

**本项目不写入**：仅项目相关的约定写入项目 MEMORY.md。

## 4. 会话交接机制

### 4.1 新会话启动时

AI 应：
1. 自动读取 `.workbuddy/memory/MEMORY.md` 和最近 3 天的每日日志
2. 读取项目根目录下的 AI 提示词文档
3. 快速扫描 `package.json`、关键配置文件、src 目录结构
4. 若用户说「继续昨天的任务」，则重点查看最近日志中的未完成项

### 4.2 用户重新进入项目时

AI 应：
1. 不假设用户记得所有上下文
2. 用 1-2 句话总结当前项目状态
3. 询问用户本次要处理的具体任务

### 4.3 多轮需求变更时

AI 应：
1. 每次变更后立即更新 MEMORY.md / 每日日志
2. 对互相冲突的历史决策进行标注（如「已废弃：旧导航栏透明方案」）
3. 重要决策（如技术栈、品牌色）变更前与用户确认

## 5. 文档更新流程

当发生以下变更时，必须同步更新对应文档：

| 变更类型 | 必须更新的文档 |
|----------|---------------|
| 新增数据表/字段 | `supabase/schema.sql`、所有相关 `migration-*.sql`、`MEMORY.md` 数据模型部分 |
| 新增后台管理模块 | `README.md` 功能列表、`MEMORY.md` 已交付部分 |
| 新增/修改图片上传功能 | 对应管理页面组件、前台展示组件、测试脚本 |
| 视觉/品牌调整 | `MEMORY.md` 品牌规范、`广州索然电梯配件官网-AI提示词.md` 视觉指导部分 |
| 部署配置变更 | `README.md`、`.github/workflows/deploy.yml`、`next.config.js` |
| 发现安全/性能问题 | `代码审查报告.md`、更新 `MEMORY.md` 待办 |

## 6. AI 提示词管理

### 6.1 提示词文档作用

`广州索然电梯配件官网-AI提示词.md` 是项目的「AI 执行说明书」，应包含：
- 网站定位与目标受众
- 页面结构与内容要求
- 内容风格与视觉指导
- 移动端适配要求
- 技术架构与部署规范
- 数据模型与前后端契约
- AI 执行规则

### 6.2 提示词更新时机

- 每次重大需求变更后，若提示词中的描述已过时，应同步修订
- 新增后台模块、数据表后，补充前后端数据契约表
- 用户明确调整风格/定位后，更新相关内容章节

### 6.3 复用提示词

后续新会话可直接引用该提示词作为项目背景输入，减少重复说明。

## 7. 代码变更追踪

### 7.1 任务管理

- 使用 TaskCreate / TaskUpdate / TaskList 跟踪每个功能/修复
- 复杂任务拆分为小任务，逐个完成并标记
- 任务描述中注明影响的文件和验收标准

### 7.2 测试与验证

- 每次代码变更后运行 `npm run build`
- 关键功能变更后运行对应测试脚本
- 数据库变更后使用 Node.js 脚本验证连接和权限

### 7.3 废弃文件清理

- 发现未使用文件（如之前的 `ProductDetailContent.jsx`）应及时删除
- 删除前确认无其他文件引用

## 8. 上线后运维交接

### 8.1 必须保留的上下文

上线后，AI 应记住：
- 项目已部署到哪个 GitHub 仓库和域名
- Supabase 项目地址
- 管理员账号邮箱（不保存密码）
- 当前 Next.js 版本及安全漏洞状态
- 已知待优化项

### 8.2 后续迭代流程

1. 用户提出新需求
2. AI 读取 MEMORY.md 和日志，确认当前状态
3. AI 评估需求影响范围（是否需要改数据库、改前台、改后台）
4. 制定计划并与用户确认
5. 执行 → 测试 → 更新文档 → 更新记忆

---

## 9. 关键原则

1. **不捏造**：所有记录必须基于实际发生的工作，不虚构功能状态。
2. **及时更新**：完成工作后立即写记忆，不要累积到最后。
3. **分层管理**：项目级记忆放 MEMORY.md，当天细节放每日日志，个人偏好放 ~/.workbuddy/MEMORY.md。
4. **可追溯**：每次重要变更都要能通过日志找到原因和时间。
5. **可验证**：文档中的状态应能通过构建、测试脚本或页面访问验证。
